## 2. Главный вывод

### Можно ли запускать сейчас?

**Нет. Три блокера:**

1. **`NEXT_PUBLIC_SITE_URL=http://localhost:3000`** — весь SEO (canonical, OG, sitemap, JSON-LD) работает на localhost. Google и Яндекс будут индексировать неправильные URL. Это критично: 30 минут в `prod.env`, ноль строк кода.

2. **Telegram-уведомления не настроены** — лиды с формы не приходят в мессенджер, только открывается Telegram с текстом. Если пользователь закрыл вкладку раньше — заявка потеряна.

3. **Пароли в `.env.local` — дефолтные** — `Printlab2024!secure` и `s3cur3-printlab-s3ss10n-s3cr3t-k3y` опубликованы в коде этой сессии. Перед деплоем обязательно менять.

После исправления трёх пунктов — **можно запускать**.

## 3. P0 — Критично до production

### P0-1: NEXT_PUBLIC_SITE_URL указывает на localhost

**Файл:** `.env.local`  
**Проблема:** `NEXT_PUBLIC_SITE_URL=http://localhost:3000`. Все canonical URL, OG `url`, sitemap, JSON-LD `@id` и `url` содержат `http://localhost:3000/...`.  
**Почему плохо:** Google/Яндекс увидят canonical на localhost — страницы не войдут в индекс под правильным доменом. OG-ссылки в соцсетях ведут в никуда.  
**Как исправить:** В production environment выставить `NEXT_PUBLIC_SITE_URL=https://ваш-домен.ru`.  
**Риск:** Полный провал SEO при запуске. Индексация localhost вместо реального домена.

---

### P0-2: Telegram-бот не настроен — лиды теряются

**Файл:** `.env.local` + `src/app/api/lead/route.ts`  
**Проблема:** `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` пусты. При отправке формы открывается Telegram-диалог, но если пользователь его закрыл или не отправил — заявка не сохраняется нигде (нет БД, нет email, нет лога).  
**Почему плохо:** Первые же реальные лиды могут быть потеряны бесследно. API отвечает `{ok:true, delivered:false}`, но фронт показывает «Заявка принята!».  
**Как исправить:** Создать Telegram-бота через @BotFather, получить chat_id через @userinfobot, заполнить env. Дополнительно: добавить логирование заявок в файл или базу как резерв.  
**Риск:** Потеря лидов с первого дня. Клиент думает, что оставил заявку — менеджер ничего не получил.

---

### P0-3: Дефолтные секреты в `.env.local`

**Файл:** `.env.local`  
**Проблема:** `ADMIN_PASSWORD=Printlab2024!secure` и `ADMIN_SESSION_SECRET=s3cur3-printlab-s3ss10n-s3cr3t-k3y` — значения присутствовали в истории сессии и могут быть известны.  
**Почему плохо:** Если эти значения попадут в репозиторий или в чужие руки — полный доступ к admin-панели и всем CMS-данным.  
**Как исправить:** Сгенерировать новые значения: `openssl rand -base64 32`. Убедиться, что `.env.local` в `.gitignore`.  
**Риск:** Компрометация admin-панели, удаление товаров, подмена контента.

---

### P0-4: Отсутствует Политика конфиденциальности

**Файл:** Страница не существует  
**Проблема:** Форма заявки (`LeadForm`) собирает имя и телефон. В форме написано «вы соглашаетесь на обработку данных» — но ссылки на документ нет.  
**Почему плохо:** Нарушение 152-ФЗ «О персональных данных». При жалобе — штраф и предписание.  
**Как исправить:** Создать страницу `/privacy` с политикой. В форме добавить ссылку на неё.  
**Риск:** Юридический риск, недоверие со стороны осторожных клиентов.

---

### P0-5: `window?.location?.origin` в server-side коде

**Файл:** `src/lib/analytics-repository.ts`, строка 106  
**Проблема:** `if (e.referrer && e.referrer !== window?.location?.origin)` — `window` недоступен в Node.js. Хотя `?.` не бросает ошибку, referrer-фильтрация работает некорректно на сервере (всегда `undefined !== referrer`).  
**Почему плохо:** Все referrer-записи пройдут фильтр — реферальная статистика будет завышена. При неудачной сборке может упасть.  
**Как исправить:** Заменить на `siteConfig.url` или передавать origin как параметр.  
**Риск:** Неправильная аналитика по источникам трафика.

---

### P0-6: `/admin` не закрыт от роботов

**Файл:** `src/app/robots.ts`  
**Проблема:** В `disallow` указано только `/api/`. Путь `/admin` открыт для индексации.  
**Почему плохо:** Google/Яндекс могут проиндексировать страницу логина, что нежелательно.  
**Как исправить:** Добавить `/admin` в `disallow`.  
**Риск:** Индексация страницы авторизации, видимость для сканеров.

## 6. Что удалить или упростить

### 6.1 `data/analytics.json` — не коммитить в git

Файл создаётся в runtime и может вырасти до 50 000 событий (~50+ МБ). Должен быть в `.gitignore`.

### 6.2 Дублирование кнопок «сторона печати»

В `Configurator.tsx` раздел «Загрузка» содержит повторный grid из 2 кнопок «Перед» / «Спина» (строки 305–324), идентичный разделу «Сторона печати» (строки 208–240). Пользователя это запутывает.

### 6.3 `src/components/ui/Icon.tsx`

Файл существует, но `Icon` не импортируется ни в одном компоненте — мёртвый код.

### 6.4 Manual-отзывы без реальных данных

3 из 6 отзывов в `reviews.ts` — придуманные. Показывать придуманные отзывы рядом с реальными Яндекс.Картами — риск репутации. Удалить или заменить.

### 6.5 `ReviewSource` тип «avito» — не используется реально

Avito-отзывов нет, но тип и фильтрация по нему есть. После заполнения реальных данных убрать заглушки.

## 7. Что оставить как сильную сторону

| Компонент | Почему хорошо |
|---|---|
| **Server Components по умолчанию** | LCP быстрый, нет гидрационной задержки для контента |
| **JSON-LD полный комплект** | LocalBusiness + Product + FAQ + Review + Breadcrumb + Website — готово к расширенным сниппетам Google |
| **`siteConfig` — единая точка контактов** | Телефон, адрес, мессенджеры — меняется в одном месте |
| **Honey-pot в форме** | Простая защита от ботов без CAPTCHA |
| **HMAC-подпись сессии** | Безопасная авторизация без сессионной БД |
| **Atomic write (rename) в репозиториях** | Нет риска повреждения JSON при одновременных запросах |
| **Mutation queue** | Thread-safe запись без race conditions |
| **File signature validation при загрузке изображений** | Защита от загрузки вредоносных файлов |
| **`prefers-reduced-motion`** | Все анимации уважают доступность |
| **`aria-*` и `role`** | Конфигуратор и диалоги доступны для скринридеров |
| **Мобильный sticky CTA** | Появляется после 500px скролла — не мешает, но всегда доступен |
| **Rate limiting на /api/lead** | 5 запросов за 10 минут с одного IP |
| **WebP + next/image** | Автоматическая оптимизация, `sizes` заданы корректно |
| **`force-dynamic` на страницах с управляемым контентом** | Свежий контент без stale cache |

## 8. SEO-решение

### Сравнение вариантов

| Вариант | SEO | Скорость запуска | Риски |
|---|---|---|---|
| Vite SPA (если бы был) | ❌ Плохо — JS не индексируется Яндексом | Быстро | Критично для SEO |
| Vite + prerender | ⚠️ Средне — статические HTML, но нет динамики | Средне | Ограничения для динамического контента |
| **Текущий: Next.js App Router SSR** | ✅ Отлично — SSR, все метатеги на сервере | Уже готово | Нет (правильный выбор) |
| Отдельный Next.js site/ | Избыточно — уже на Next.js | Долго | Дублирование кодовой базы |

### Финальная рекомендация

**Проект уже на Next.js — это правильное решение. Переносить ничего не нужно.**

Текущий стек закрывает все SEO-требования:
- SSR на сервере → H1, title, description в HTML до JavaScript
- Автоматический `canonical` через `alternates`
- JSON-LD рендерится серверно через `<JsonLd>`
- `sitemap.xml` и `robots.txt` генерируются Next.js API
- `generateMetadata` на каждом маршруте

**Единственное действие для SEO:** исправить `NEXT_PUBLIC_SITE_URL` на реальный домен.

### SEO-риски после запуска

1. Страницы `/product/[slug]` имеют `dynamic = "force-dynamic"` — Яндекс любит стабильные URL, но это не критично.
2. `sitemap.xml` перечисляет товары с `priority: 0.7` — без `lastModified` на уровне продукта (только `new Date()`) Яндекс не знает, что обновилось.
3. Нет hreflang — если планируется международный трафик, добавить.

## 9. UX/CRO-рекомендации

### 9.1 Путь пользователя

```
Hero → Категории → Карточки товаров → Страница товара → Заказ в WhatsApp
                                    ↘ Конструктор → Оформить заказ → WhatsApp/Telegram
Hero → LeadCta (форма) → Telegram
```

**Проблема:** Путь длинный. От первого экрана до реальной заявки — минимум 3–4 действия. Нет «быстрого» пути для тех, кто уже знает, что хочет.

### 9.2 Рекомендации по конверсии

| # | Рекомендация | Приоритет |
|---|---|---|
| CRO-1 | Добавить номер телефона в Hero-секцию (не только в шапку) — телефон = доверие | Высокий |
| CRO-2 | В конструкторе добавить шаг «Укажите телефон для связи» прямо в форму диалога — не открывать внешний мессенджер | Высокий |
