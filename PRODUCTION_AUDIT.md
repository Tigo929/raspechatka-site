# Производственный аудит «Распечатки»

Актуально на 12 июня 2026 года.

## Итог

Проект приведён к одной рабочей архитектуре: Next.js обслуживает публичный сайт,
SEO-страницы, конструктор, API заявок и защищённую админ-панель. Незавершённый и
неподключённый NestJS-контур удалён, чтобы не поддерживать две расходящиеся модели данных.

Проверки текущего состояния:

- `npm run typecheck` — успешно;
- `npm run lint` — успешно;
- `npm run test` — 3 теста успешно;
- `npm run build` — успешно, 45 маршрутов;
- desktop/mobile QA — нет горизонтального переполнения, mockup и зона печати отображаются;
- `npm audit` — известных уязвимостей нет.

## Надёжность заявок

Заявка больше не считается успешной только потому, что API Telegram ответил ошибкой.
Новый порядок обработки:

1. сервер валидирует контакт, согласия и файлы;
2. заявка атомарно записывается в `orders.json`;
3. оригиналы и превью макетов сохраняются рядом с заявкой;
4. только после сохранения выполняется доставка в Telegram;
5. при недоступности Telegram заявка получает статус `failed`, но не теряется;
6. в `/admin` видны все обращения и доступна повторная отправка.

Клиент показывает успех только при `stored: true` и выводит номер обращения вида
`RP-20260612-ABC123`.

## Конструктор

- Белая и чёрная футболки связаны с соответствующими mockup-фото.
- Перед и спина имеют независимые изображения и независимые координаты.
- Пользователь всегда видит границу зоны печати.
- В заказ передаются оригиналы, координаты, масштаб и PNG-превью обеих сторон.
- До явной отправки изображения обрабатываются только в браузере.
- Лимит клиента и сервера унифицирован: 10 МБ на изображение.

## Данные и админка

- Контакты из настроек админки используются в Header, Footer, мобильном CTA,
  контактах и Schema.org.
- Категории админки используются публичным каталогом и Footer.
- JSON-файлы записываются атомарно; конкурирующие мутации сериализованы.
- Загруженные JPG/PNG/WebP действительно конвертируются в формат целевого файла.
- Тестовые ручные отзывы не публикуются на сайте; в админке они сохранены.

## Аналитика и согласия

- Внутренняя аналитика запускается только после согласия пользователя.
- Изменение согласия применяется без перезагрузки страницы.
- События пишутся в append-only `analytics.ndjson`, а не перезаписывают весь JSON.
- Исправлен серверный расчёт referrer, который раньше обращался к `window`.
- Runtime-аналитика исключена из Git.

## Перед публикацией

Обязательные действия инфраструктуры:

1. Задать уникальные `ADMIN_PASSWORD` и `ADMIN_SESSION_SECRET`.
2. Подключить постоянный диск и указать `PRINTLAB_DATA_DIR`.
3. Задать `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`.
4. Настроить резервное копирование каталога `PRINTLAB_DATA_DIR` и
   `public/uploads/products`.
5. Заполнить юридические заглушки перед приёмом оплаты и публичным запуском.
6. Выполнить тестовую заявку и проверить её появление в `/admin` и Telegram.

Без постоянного диска файловое хранилище нельзя размещать на serverless-инстансе с
эфемерной файловой системой. В такой среде следует подключить managed PostgreSQL и
S3-совместимое объектное хранилище через текущий repository-слой.
